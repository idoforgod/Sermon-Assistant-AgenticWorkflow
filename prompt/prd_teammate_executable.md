## Fork-Based Sessions Branch Strategy (확장 기능)

이 프레임워크는 `/fork` 기능을 활용하여 **각 Phase에서 여러 경로를 동시에 탐색**합니다.
이를 통해 더 깊이 있는 리서치, 철저한 토론, 견고한 수렴을 이룹니다.

### 전략 개요

```
초기 설정
  ↓
[PHASE 1: 조사]
  ├─ /fork Branch 1.1: Market Researcher (기회주의 관점)
  ├─ /fork Branch 1.2: Market Researcher (보수주의 관점)
  ├─ /fork Branch 2.1: User Researcher (극단적 사용자)
  ├─ /fork Branch 2.2: User Researcher (평균적 사용자)
  ├─ /fork Branch 3.1: Tech Architect (모놀리식 설계)
  ├─ /fork Branch 3.2: Tech Architect (마이크로서비스 설계)
  ├─ /fork Branch 4.1: Business Strategist (공격적 성장)
  └─ /fork Branch 4.2: Business Strategist (안정적 성장)
  ↓
[PHASE 2: 토론]
  ├─ /fork Branch 2.A: 시장 기회 우선 토론
  ├─ /fork Branch 2.B: 사용자 만족 우선 토론
  ├─ /fork Branch 2.C: 기술 실현성 우선 토론
  └─ /fork Branch 2.D: 비즈니스 지속성 우선 토론
  ↓
[PHASE 3: 수렴]
  ├─ /fork Branch 3.A: Aggressive Scenario (위험도 높음, 기회도 큼)
  ├─ /fork Branch 3.B: Balanced Scenario (위험-기회 균형)
  └─ /fork Branch 3.C: Conservative Scenario (위험도 낮음, 확실성 높음)
  ↓
[PHASE 4: 통합 및 최종 결정]
  → 3개 시나리오 비교 → 최적 경로 선택
```

### Branch 사용의 목적

**조사 Phase에서의 Branch**
- 같은 관점에서도 다양한 가정이 가능함을 인정
- 예: "시장이 크다고 본다" vs "시장이 작다고 본다"
- 각 가정이 타당한 근거를 가질 수 있음을 검증

**토론 Phase에서의 Branch**
- 어떤 관점을 가장 우선시하냐에 따라 결론이 달라짐을 검증
- "시장 우선" vs "사용자 우선" vs "기술 우선" vs "비즈니스 우선"의 각 경로가 다른 PRD를 만듦
- 각 경로의 장단점 파악

**수렴 Phase에서의 Branch**
- 공격적 vs 균형 vs 보수적 3가지 시나리오 동시 검증
- "만약 우리가 더 적극적이었다면?" "더 신중했다면?"을 동시에 평가
- 최종 선택의 근거를 명확히 함

## 실행 제약 (Constraints)

이 프로세스를 진행할 때 다음 제약을 유지하세요:

**조사 단계 제약**
- 각 Teammate는 **근거 없는 주장 금지** (추측/직관만으로는 안 됨)
- 기존 경쟁사/유사 사례 최소 3개 언급
- "불가능하다"는 결론은 구체적 이유 3가지 이상 제시
- 각 Teammate는 다른 관점의 영향을 받지 말 것 (독립성 유지)

**토론 단계 제약**
- Moderator는 중립 입장 유지 (특정 팀의 편 들지 않기)
- 한 관점이 연속 3번 이상 강조하면, 반대 관점에 명시적 발언 기회 제공
- "동의한다"만으로는 안 됨 → "왜 동의하는가" 근거 포함 필수
- 충돌이 없으면 안 됨 (4개 관점이 모두 일치하면 토론 깊이가 부족한 신호)

**수렴 단계 제약**
- Green Zone: **4개 관점 모두 동의** (3개만 동의하면 Yellow Zone)
- Yellow Zone: **명확한 트리거 조건** 정의 필수 ("언제 포함될 것인가?")
- Red Zone: **재검토 시점 명시** ("6개월 후" 또는 "~이면 검토")
- 기능은 최대 15개 이내로 집약 (산만해지면 안 됨)

## 성공 지표 (Success Points)

이 PRD 프로세스가 **성공했는지** 판단하는 기준:

### Phase 1 (조사) 성공 지표
✅ **각 Teammate별 결론이 구체적인가?**
- [ ] 기능명이 구체적인가 (예: "대시보드" ❌ → "매일 아침 10시 업데이트되는 주간 매출 대시보드" ✅)
- [ ] 근거가 명시되어 있는가 (예: "경쟁사 3개 분석 결과...", "사용자 인터뷰 5명 중 4명...")
- [ ] 우선순위가 명확한가 (Top 3이 명확히 도출됨)

✅ **4개 관점이 서로 다른 각도에서 봤는가?**
- [ ] Market만 강조하는 기능이 있는가? (시장 기회만 보는 것)
- [ ] User만 강조하는 기능이 있는가? (사용성만 보는 것)
- [ ] Tech만 조심하는 기능이 있는가? (기술 리스크)
- [ ] Business가 우려하는 기능이 있는가? (수익성 문제)
- → 모두 있으면 ✅ 다각적 분석 성공

### Phase 2 (토론) 성공 지표
✅ **충돌이 명확히 드러났는가?**
- [ ] 최소 3개 이상의 의견 불일치 포착됨
- [ ] 각 불일치마다 "왜 다른가"의 근거가 명시됨
- [ ] 단순 선호도 차이가 아니라 **논리적 충돌**임

✅ **합의 영역이 명확해졌는가?**
- [ ] 4개 관점이 **모두 동의**하는 기능이 최소 5개 이상
- [ ] 그 기능들이 핵심 고객 문제를 **정말 해결**하는가?
- [ ] Green Zone 기능들이 "이것들만으로도 MVP가 되나?"의 답이 "YES"

✅ **위험 가정이 포착되었는가?**
- [ ] Top 5 위험 가정이 명시됨
- [ ] 각 가정마다 "검증 방법"이 구체적임 (예: "사용자 인터뷰 10명", "경쟁사 벤치마킹")
- [ ] 위험 가정이 "비즈니스를 망칠 수 있는" 수준의 것들임

### Phase 3 (수렴) 성공 지표
✅ **PRD가 실행 가능한가?**
- [ ] Green Zone 기능들의 합계 개발 시간이 명시되고 **현실적**인가?
- [ ] 각 기능의 우선순위 순서가 명확한가?
- [ ] Phase 2로 미룬 기능들이 **정말 미룰 수 있는** 것들인가? (필수는 아닌가?)

✅ **모든 팀이 서명할 수 있는가?**
- [ ] Market Analyst: "이 PRD로 시장 기회를 잡을 수 있다" → 동의하는가?
- [ ] User Researcher: "이 기능들로 사용자 문제가 해결되는가?" → 동의하는가?
- [ ] Tech Architect: "6개월에 이 정도는 만들 수 있는가?" → 동의하는가?
- [ ] Business Strategist: "이 조합으로 돈을 벌 수 있는가?" → 동의하는가?
- → 모두 "동의"해야 ✅

✅ **미래에 후회하지 않을 결정인가?**
- [ ] 왜 이 기능을 택했고 왜 뺐는지가 명시되어 있는가?
- [ ] 만약 이 가정이 틀렸을 때 어떻게 할 것인가?
- [ ] 6개월 후 "왜 이걸 한 거야?"라는 물음에 답할 수 있는가?

## 최종 검수 체크리스트

PRD가 완성되었을 때 이것을 확인하세요:

```
조사 단계:
[ ] Market Researcher 결론이 구체적인가?
[ ] User Researcher의 페르소나 정의가 생생한가?
[ ] Tech Architect의 추정이 보수적인가?
[ ] Business Strategist의 수익화 논리가 타당한가?

토론 단계:
[ ] 최소 3개 이상의 의견 불일치가 기록되어 있는가?
[ ] 각 불일치에 대한 해결책 또는 우선순위가 결정되어 있는가?
[ ] Top 5 위험 가정과 검증 계획이 명시되어 있는가?

수렴 단계:
[ ] Green Zone 기능이 5-7개로 집약되어 있는가?
[ ] 각 Green Zone 기능이 "반드시 필요"한 이유가 명시되어 있는가?
[ ] Yellow Zone 기능에 "포함 조건"이 명시되어 있는가?
[ ] Red Zone 기능에 "재검토 시점"이 명시되어 있는가?

최종:
[ ] 4명의 Teammate 모두가 이 PRD에 동의할 것 같은가?
[ ] 6개월 후 "우리가 한 선택이 맞았다"고 판단할 것 같은가?
[ ] 만약 PRD를 벗어나는 요청이 들어오면, 이 문서로 거절할 수 있는가?
```

## PHASE 1 Execution: Fork-Based 병렬 조사 구조

### 단계 1: 각 Teammate의 2개 Branch 독립 실행

이 단계에서는 **각 Teammate가 자신의 관점 내에서 상충하는 2가지 가정**을 동시에 탐색합니다.

#### **Market Researcher Branch A: Optimistic View (기회주의)**

```
당신은 이 제품의 시장 기회를 **최대한 긍정적으로** 분석하는 관점입니다.

핵심 가정: "이 시장은 빠르게 성장하고 있으며, 지금 진입하면 성공할 수 있다"

분석 내용:
1. 시장 규모: [가능한 한 크게 추정]
   - 연간 성장률: [높음]
   - 우리가 잡을 수 있는 규모: [큼]

2. 경쟁 상황: [기존 경쟁사의 약점 강조]
   - 경쟁사 3-5개의 약점: [구체적]
   - 우리의 명확한 차별화: [3가지]

3. 진입 타이밍: [지금이 황금기]
   - 왜 지금인가? [구체적 신호들]
   - 늦으면 어떻게 되나? [손실]

결론:
- 이 기회를 놓치면: [얼마나 손해인가?]
- 반드시 포함되어야 할 기능: [Top 3]
- 이 기능들이 없으면: [경쟁에서 졌다]
- 위험 요소: [하지만 이 정도면 관리 가능]
```

#### **Market Researcher Branch B: Cautious View (보수주의)**

```
당신은 이 제품의 시장을 **신중하고 비판적으로** 분석하는 관점입니다.

핵심 가정: "이 시장이 생각보다 작거나 이미 포화되어 있을 수 있다"

분석 내용:
1. 시장 규모: [보수적으로 추정]
   - 연간 성장률: [낮음]
   - 우리가 현실적으로 잡을 수 있는 규모: [작음]

2. 경쟁 상황: [기존 경쟁사의 강점 강조]
   - 경쟁사 3-5개의 강점: [구체적]
   - 우리가 진정으로 다른 부분: [3가지]

3. 진입 타이밍: [너무 늦지 않았는가?]
   - 이미 진입한 경쟁사들: [구체적]
   - 우리가 이길 방법: [있는가?]

결론:
- 현실적인 시장 규모: [구체적 수치]
- 반드시 포함되어야 할 기능: [Top 3]
- 이 기능들이 없으면: [차별화 불가]
- 주요 리스크: [심각한 3가지]
```

#### **최종 정리 (Branch A vs B 비교)**
```
정성적 근거:
- Optimistic이 타당한 신호: [구체적 증거]
- Cautious가 타당한 신호: [구체적 증거]

데이터 기반 판단:
- 현재 시장 데이터는 어느 쪽을 더 지지하는가?
- 우리가 놓친 데이터는 무엇인가?
- 다음 2주에 수집해야 할 데이터: [구체적]
```

---

#### **User Researcher Branch A: Edge Case (극단적 니치 사용자)**

```
당신은 **이 제품이 정말 필요한 극단적 사용자**를 타겟으로 분석하는 관점입니다.

페르소나: "이 제품이 없으면 그들의 삶이 크게 불편해지는 사람"

분석 내용:
1. 페르소나 상세 분석
   - 이름, 역할, 배경: [매우 상세히]
   - 현재 고통: [존재하지 않으면 못 견딜 정도]

2. 그들이 정말로 필요한 기능
   - Top 3 기능: [이 없으면 쓸 가치 없음]
   - 각 기능이 해결하는 구체적 문제: [명확히]

3. 성공의 정의
   - "Edge Case 사용자 100명이 우리를 쓰면": [성공]
   - 그들의 만족도 목표: [구체적 지표]

결론:
- 이 페르소나를 위한 핵심 기능 3가지: [구체적]
- 이 기능들이 없으면: [이 사용자들은 안 온다]
```

#### **User Researcher Branch B: Mainstream (평균적 대중 사용자)**

```
당신은 **평균적인 잠재 사용자**를 타겟으로 분석하는 관점입니다.

페르소나: "이 제품이 있으면 좋지만, 없어도 버틸 수 있는 사람"

분석 내용:
1. 페르소나 상세 분석
   - 이름, 역할, 배경: [일반적인 상황]
   - 현재 불편함: [있으면 해결, 없어도 대안 있음]

2. 그들이 원하는 기능
   - Top 3 기능: [있으면 쓸 만함]
   - 각 기능의 우선순위: [명확히]

3. 성공의 정의
   - "Mainstream 사용자 10,000명이 우리를 쓰면": [성공]
   - 그들의 활성도 목표: [구체적 지표]

결론:
- 이 페르소나를 위한 핵심 기능 3가지: [구체적]
- 이 기능들이 없어도: [대안이 있어서 괜찮음]
```

#### **최종 정리 (Branch A vs B 비교)**
```
타겟 선택의 의미:
- Edge Case 우선이면: [개발 방향, 마케팅 방향]
- Mainstream 우선이면: [개발 방향, 마케팅 방향]

현실적 질문:
- 둘 다 만족시킬 수 있는가?
- 만족시킬 수 없다면, 누구를 버릴 것인가?
- 버린 그룹은 언제 다시 돌아올 것인가? (Phase 2?)
```

#### **Tech Architect Branch A: Monolithic (초기 빠른 출시)**

```
당신은 **초기 빠른 출시**를 우선하는 기술 전략을 분석하는 관점입니다.

철학: "처음엔 빠르고 단순하게, 나중에 리팩토링한다"

분석 내용:
1. 아키텍처 전략
   - 구조: [모놀리식 구조]
   - 기술 스택: [빨리 만들 수 있는 것들]
   - 복잡도: [낮음]

2. 개발 속도
   - 6개월 내 구현 가능 범위: [매우 넓음]
   - 각 기능별 개발 시간: [구체적]

3. 기술 부채
   - 초기에 쌓일 부채: [구체적]
   - 추후 리팩토링 비용: [얼마나 들 것인가?]

결론:
- 6개월에 이 정도는 확실히 만들 수 있음: [리스트]
- 초기 기술 부채 위험: [높음/중간/낮음]
- 추후 리팩토링 계획: [언제? 얼마나?]
```

#### **Tech Architect Branch B: Microservices (장기 확장성)**

```
당신은 **장기 확장성**을 우선하는 기술 전략을 분석하는 관점입니다.

철학: "초기부터 견고한 구조로, 확장에 대비한다"

분석 내용:
1. 아키텍처 전략
   - 구조: [마이크로서비스 구조]
   - 기술 스택: [확장 가능한 것들]

   - 돈을 못 벌면 우리는 없다

3. 비즈니스 리스크를 최소화하면서 성장

당신은 3개 Scenario를 받았습니다.

지금의 임무:
1. 각 Scenario의 본질 파악
2. 우리 상황에 맞는 Scenario 선택
3. 선택의 근거 기록
4. 최종 PRD 확정

---

## STEP 1: 3개 Scenario 재검토

### Aggressive를 택할 조건:
- [ ] 시장 긴급성이 높다 (경쟁사가 빨리 나온다)
- [ ] 우리 팀의 기술력이 탁월하다
- [ ] 실패해도 회사가 죽지 않는다
- [ ] 창립자/투자자가 공격적이다

→ 3개 이상 해당하면 Aggressive

### Balanced를 택할 조건:
- [ ] 시장 긴급성이 중간 정도다
- [ ] 우리 팀이 균형잡혀 있다
- [ ] 단기 성공도 중요하고 장기 지속도 중요하다
- [ ] 현실적인 접근을 선호한다

→ 3개 이상 해당하면 Balanced

### Conservative를 택할 조건:
- [ ] 시장이 충분히 크고 빠르지 않다
- [ ] 불확실성이 높다 (사용자/기술/시장)
- [ ] 높은 품질과 안정성이 경쟁력이다
- [ ] 장기 관점에서 우리가 생존하는 것이 중요하다

→ 3개 이상 해당하면 Conservative

## STEP 2: 선택된 Scenario의 PRD를 최종 확정

확인 사항:
- [ ] 이 PRD가 4개 Perspective를 충분히 담고 있는가?
- [ ] 이 PRD가 우리의 상황과 맞는가?
- [ ] 6개월 후 "이 선택이 맞았다"고 할 수 있을 것 같은가?

## STEP 3: 최종 검증 (4명 Teammate의 서명)

각 Teammate에게:

```
"당신의 관점에서 봤을 때, 이 최종 PRD가 옳다고 생각하는가?"

조건:
- "동의합니다" = 당신의 핵심 관점이 담겨 있다
- "수용합니다" = 완벽하지 않지만 현실적이다
- "우려합니다" = 당신의 관점이 충분히 반영되지 않았다

각각의 의견과 근거를 명시하세요.
```

최종 서명:
```
✅ Market Researcher: [동의/수용/우려] 근거: [구체적]
✅ User Researcher: [동의/수용/우려] 근거: [구체적]
✅ Tech Architect: [동의/수용/우려] 근거: [구체적]
✅ Business Strategist: [동의/수용/우려] 근거: [구체적]

결과:
- 4개 동의: 이상적 (최고의 PRD)
- 4개 수용: 현실적 (실행 가능한 PRD)
- 1개 이상 우려: 주의 (우려 사항 추가 검토 필요)
```

## STEP 4: 최종 결과물 확정

PRD 확정과 함께 제공할 것:

1. **최종 PRD** (Green/Yellow/Red Zone 명시)
2. **의사결정 기록** (왜 이 Scenario를 택했는가?)
3. **위험 관리 계획** (Top 5 가정, 검증 방법, 대응책)
4. **팀 정렬 문서** (모든 Teammate의 서명)
5. **Branch 비교 분석** (3개 Scenario의 차이와 의미)

→ 이것들이 당신의 최종 산출물입니다.

## 팁 & Best Practices

### Branch 전략을 효과적으로 운영하기

**PHASE 1 (조사 Branch) 운영 팁**
- 각 Branch는 **완전히 독립적**이어야 함 (다른 Branch의 결과를 보지 말 것)
- 기회주의 vs 보수주의, 극단적 vs 평균적 등의 **상충하는 가정**을 명확히 할 것
- 모든 결론에 **근거(데이터, 사례, 인터뷰)**를 첨부할 것
- "모든 Branch가 동의하는 기능"을 찾을 것 (그것이 진정한 필수 기능)

**PHASE 2 (토론 Branch) 운영 팁**
- 각 Branch는 **특정 관점을 극단까지 밀어본다** (시장 우선이면 정말 시장만 봄)
- 이렇게 하면 "어떤 관점을 1순위로 할 때 어떤 결과가 나오는가"를 명확히 봄
- 4개 Branch의 결과를 비교할 때 "기능별 우선순위 변화"를 표로 정리할 것
- "어떤 관점 조합이 이상적인가?"를 발견할 것

**PHASE 3 (시나리오 Branch) 운영 팁**
- Aggressive, Balanced, Conservative는 **단순히 보수/공격이 아니라 철학적 선택**
- 각 시나리오는 "우리가 이 길을 가면 6개월 후 이렇게 되어 있을 것"을 명확히 보여줄 것
- 선택 기준을 명확히 할 것 (시장 상황, 팀 역량, 자본, 리스크 용인도 등)

**PHASE 4 (최종 통합) 운영 팁**
- 3개 Scenario 중에서 "1등"을 고르는 것이 아니라, "우리 상황에 맞는" 것을 고르기
- "이 Scenario를 택하지 않은 이유는?"를 기록할 것 (추후 의사결정의 근거)
- 최종 PRD에 "선택 근거"와 "버린 시나리오"를 명시할 것 (미래의 당신이 이해하게)

### 원래 있던 팁들 (유지)

조사 단계에서
- 각 Teammate가 "근거"를 명확히 하도록 강조
- 추상적 답변은 "구체적 예시를 3가지 주세요"로 재요청
- 충돌하는 의견이 나오면 좋음 (건강한 신호)

토론 단계에서
- Moderator가 중립적이어야 함
- 한 팀의 의견이 독주하면 "다른 관점은?"으로 균형 조절
- "동의합니다"는 약한 신호 → "왜 동의합니까?"로 깊이 있게

수렴 단계에서
- Green Zone은 **보수적으로** (모두 동의한 것만)
- Yellow Zone은 **명확한 기준**으로 (언제 포함될 것인가?)
- Red Zone도 **언제 재검토할 것인가** 명시
